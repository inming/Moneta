# Moneta — 产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品愿景

Moneta 是一款面向个人和家庭的桌面端记账软件，帮助用户轻松记录日常收支、分类管理财务、可视化分析消费习惯。通过 AI 截图识别能力，大幅降低手动录入的成本。

### 1.2 目标用户

- 有记账习惯的个人用户（从 Excel 记账迁移而来）
- 家庭用户（多人共同记账，按操作人区分）

### 1.3 核心价值

| 价值点 | 描述 |
|--------|------|
| 高效录入 | AI 截图识别 + 批量录入，告别逐条手动输入 |
| 数据沉淀 | SQLite 本地存储，数据完全属于用户 |
| 直观分析 | 对标 Excel 透视表的交叉统计 + 可视化图表 |
| 平滑迁移 | 兼容用户现有 Excel 数据格式，无缝导入历史数据 |

---

## 2. 核心概念

| 概念 | 定义 |
|------|------|
| **交易（Transaction）** | 一条收支或投资记录，是系统的最小数据单元 |
| **类型（Type）** | 交易的性质：消费、收入 或 投资。投资用于区分非消费性支出（如基金、股票等） |
| **分类（Category）** | 交易的归类标签，消费、收入和投资各有独立分类体系 |
| **操作人（Operator）** | 记录这笔账的家庭成员，简单文本标识 |
| **记账日期** | 每次开始记账时选定的日期，本次会话所有新增条目默认使用该日期 |

## 3. 功能模块

### 3.1 交易管理

#### 3.1.1 交易数据结构

对齐用户现有 Excel 表结构：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 日期 | Date | ✅ | 默认为当前「记账日期」，可单条修改 |
| 添加时间 | DateTime | 系统生成 | 记录该条交易写入系统的时间，精确到秒，用于排序和审计，不可手动编辑 |
| 类型 | Enum | ✅ | "消费"、"收入" 或 "投资" |
| 金额 | Number | ✅ | 可为正数或负数，精确到分（两位小数）；负数用于退款、冲正等场景 |
| 分类 | String | ✅ | 从分类列表中选择 |
| 描述 | String | ❌ | 自由文本备注 |
| 操作人 | String | ❌ | 从已有操作人列表选择或输入新名称 |

#### 3.1.2 录入模式

1. **选定记账日期**：每次开始记账时，用户先选择一个日期（默认今天），本次会话中所有新增条目自动采用该日期
2. **快速录入表单**：选择类型 → 输入金额 → 选择分类 → （可选）填写描述和操作人 → 提交
3. **连续录入**：提交后表单自动清空（保留类型、日期、操作人），方便连续录入多条
4. **编辑与删除**：支持对已有交易的修改和删除

- 金额支持录入负数，用于退款、冲正等需要抵减原交易的场景

#### 3.1.3 交易列表

- 默认按日期倒序展示
- 支持按日期范围、类型、分类、操作人筛选
- 支持关键词搜索（搜索描述字段）
- 支持按添加时间排序，便于查看最新录入的数据
- 数据浏览页仅展示交易记录，分类和操作人管理已移至设置页（§3.3、§3.6）

#### 3.1.4 数据浏览与编辑

数据浏览页以单一表格形式展示所有交易记录（不含分类和操作人 Tab，这些在设置页管理）。

##### 表格结构

- 列：日期、添加时间、类型、金额、分类、描述、操作人、操作（编辑/删除按钮）
- 分页展示，支持切换每页条数（20/50/100）
- 表头显示总记录数
- 添加时间列为只读字段，显示系统实际写入该条记录的时间

##### 行内编辑

- 点击编辑按钮或双击行进入编辑状态，支持修改：
  - **日期**：日期选择器
  - **类型**：下拉选择（消费/收入/投资）
  - **金额**：数字输入框
  - **分类**：下拉选择（根据当前类型动态加载对应分类列表）
  - **描述**：文本输入框
  - **操作人**：下拉选择
- **添加时间**不可编辑，编辑已有记录时保持原值不变
- 类型变更后，若原分类不属于新类型，分类自动清空
- 编辑后保存即时写入数据库

##### 删除

- 每行操作列提供删除按钮
- 删除前弹出确认提示（防止误操作）
- 支持批量选择 + 批量删除

##### 新增一行

- 表格上方提供「新增」按钮
- 新增行各字段默认为空，但**日期字段记忆上次用户输入的值**（方便连续录入同一天的数据）
- 日期记忆仅当前会话有效，应用关闭后清空
- 新增行需填写必填字段（日期、类型、金额、分类）后方可保存

##### 按列排序

- 点击列头按该列升序/降序排序
- 支持排序的列：日期、添加时间、类型、金额、分类、操作人
- 添加时间排序基于该条记录的实际写入时间，而非记账日期
- 排序作用于全量数据结果，而非仅当前页前端重排

##### 日期范围筛选

数据浏览页工具栏支持日期范围筛选，帮助用户快速定位特定时间段的交易记录。

**筛选控件**
- **位置**：工具栏右侧，位于「搜索描述」输入框左侧
- **组件**：日期范围选择器（RangePicker）
- **格式**：YYYY-MM-DD，显示为「开始日期 ~ 结束日期」

**预设快捷选项**
点击日期选择器后下拉面板提供快捷选项：

| 选项 | 日期范围 |
|------|---------|
| 全部 | 清除日期筛选，显示全部数据 |
| 本月 | 当月 1 日 ~ 当月最后一天 |
| 上月 | 上月 1 日 ~ 上月最后一天 |
| 本季度 | 本季度首月 1 日 ~ 本季度末月最后一天 |
| 本年 | 当年 1 月 1 日 ~ 当年 12 月 31 日 |

**交互逻辑**
- 默认不限制日期范围，显示全部交易记录
- 选择日期范围后自动触发查询（防抖 300ms）
- 日期筛选与现有筛选条件（类型、分类、操作人、关键词）叠加生效
- 点击「全部」或清除按钮恢复显示全部数据
- 编辑/新增行时禁用日期筛选器（防止筛选条件变化导致编辑状态异常）

**边界处理**
- 开始日期 > 结束日期：自动交换两者顺序
- 选择的日期范围无数据：显示空状态提示「暂无符合条件的记录」

##### 从统计页返回

当用户从统计报表页通过「查看明细」功能跳转至数据浏览页后，提供快捷返回按钮，一键回到统计页并恢复之前的浏览状态。

**返回按钮显示逻辑**
- **显示条件**：当用户通过统计页趋势图的「查看明细」功能跳转过来时显示返回按钮
- **隐藏条件**：用户直接通过侧栏导航进入数据浏览页时不显示返回按钮
- **位置**：页面标题栏左侧，与「数据浏览」标题并排
- **样式**：返回箭头图标（←）+ 文字「返回统计页」，点击区域与标题保持适当间距

**状态恢复机制**
点击返回按钮后，应用导航回统计报表页（`/statistics`），并恢复以下状态：

| 状态类型 | 恢复内容 | 存储方式 |
|---------|---------|---------|
| **筛选条件** | 年份、类型（消费/收入/投资） | URL query parameters |
| **视图Tab** | 月度明细 / 年度汇总 | URL query parameters |
| **图表交互状态** | 当前独显的分类（如果有） | URL query parameters |

**URL 状态传递**
- 从统计页跳转至数据浏览页时，将当前统计页状态编码到 URL（如 `/?...&from=statistics&year=2024&type=expense&tab=monthly&soloCategory=正餐`）
- 返回时根据 `from=statistics` 判断显示返回按钮
- 返回按钮点击后，使用保存的状态重构统计页 URL（`/statistics?year=2024&type=expense&tab=monthly&soloCategory=正餐`）

**交互细节**
- 返回按钮使用浏览器历史 API 或 React Router 的 `navigate(-1)` 实现，确保返回流畅
- 若用户点击返回按钮前修改了数据浏览页的筛选条件，不影响返回功能（仅返回统计页，不携带数据浏览页的筛选状态）
- 返回后统计页的汇总卡片、交叉统计表、趋势图均基于恢复的状态重新加载数据

**边界处理**
- 用户复制带 `from=statistics` 的 URL 在新标签页打开：正常显示返回按钮，点击后正常跳转统计页
- 浏览器前进/后退操作：与返回按钮行为保持一致
- 从统计页跳转后手动修改 URL 删除 `from` 参数：返回按钮隐藏，视为直接访问数据浏览页

### 3.2 AI 图片识别

#### 3.2.1 使用场景

用户拍摄或截取以下页面的截图/照片：
- 支付宝/微信支付账单页面
- 电商平台（淘宝、京东等）订单列表
- 银行 App 交易明细
- 纸质小票、发票
- 其他任何包含交易信息的页面或凭证

#### 3.2.2 识别流程

```
┌──────────────┐    ┌───────────┐    ┌──────────────┐    ┌──────────┐    ┌────────┐
│ 上传/粘贴     │ →  │ AI 提取    │ →  │ 智能匹配分类  │ →  │ 用户确认  │ →  │ 批量入库│
│ 一张或多张图片 │    │ 多条交易   │    │ 未识别留空    │    │ 补充修改  │    │        │
└──────────────┘    └───────────┘    └──────────────┘    └──────────┘    └────────┘
```

#### 3.2.3 图片输入

- **输入方式**：支持以下两种方式提交图片：
  - 文件上传：点击上传区域或拖拽图片文件
  - 剪贴板粘贴：通过 Ctrl+V / Cmd+V 粘贴截图
- **多图支持**：一次可提交多张图片，所有图片在**单次 API 请求**中发送给 AI 一并识别
- **格式支持**：JPEG、PNG、WebP、BMP
- **大小限制**：单张图片不超过 10 MB，单次请求总量不超过 20 MB
- **图片预览**：上传后展示缩略图列表，支持删除某张图片后再发起识别

#### 3.2.4 AI 识别与分类匹配

1. **AI 提取**：调用已配置的 AI 模型（参见 §3.7 AI 模型配置），从图片中提取结构化交易数据：
   - 金额
   - 描述/商品名
   - 交易类型（消费/收入/投资）
   - 注：日期不需要 AI 提取，统一使用用户在识别页选定的「记账日期」
2. **智能分类匹配**：AI 返回结果后，系统根据每条交易的描述信息自动匹配最可能的分类：
   - 已知模式匹配（如"美团外卖"→ 正餐，"滴滴出行"→ 交通，"Steam"→ 娱乐）
   - 可利用 AI 在识别时直接建议分类（将现有分类列表作为上下文传入 prompt）
   - AI prompt 中以 JSON 数组格式传入分类列表，每项包含 `name`（分类名）和可选的 `description`（AI 描述），如：`[{"name":"住房","description":"房租、房贷月供"},{"name":"保险"}]`，避免特殊字符导致解析歧义
   - 无法识别的分类字段**留空并高亮标记**，提示用户必须手动选择
3. **分类完整性约束**：每条交易必须有分类才能提交，未填写分类的条目阻止入库

#### 3.2.5 确认界面

- 以可编辑表格形式展示所有提取的条目
- 可编辑字段：类型、金额、分类、描述、操作人
- 日期不在表格列中，由页面顶部的「记账日期」选择器统一指定，所有条目共享同一日期。**日期选择器初始为空**，用户必须主动选择日期后才能提交
- **分类为空的行高亮警示**（红色背景色），引导用户补充
- **类型颜色标记**：不同交易类型的行使用不同背景色（消费=淡橙色、收入=淡绿色、投资=淡蓝色），与系统 Tag 颜色体系一致
- 用户可逐条删除不需要的条目
- **手工插入条目**：支持在任意位置插入新的空行，方便补充 AI 未识别到的交易
  - 每行操作列提供「插入」按钮，点击后在该行**上方**插入一条空行
  - 新插入的行字段默认与当前行相同（类型、操作人），金额为空、分类为空、描述为空
  - 插入后新行自动进入编辑状态（或保持与其他行一致的编辑模式）
  - 表格底部提供「添加一行」按钮，用于在末尾追加新行
- **操作人联动**：操作人选择器**初始为空**，用户必须主动选择操作人后才能提交；选择后所有行自动同步为该操作人，后续切换操作人时所有行同步更新
- 顶部显示识别摘要：共识别 N 条，待补充分类 M 条
- **提交校验**：点击「确认录入」时检查以下必填项，任一不满足则提示用户并阻止提交：
  - 分类：所有条目的分类字段不为空
  - 日期：记账日期已选择（AI 识别为全局日期；MCP 导入为每行日期）
  - 操作人：全局操作人已选择

#### 3.2.6 批量入库

- 用户确认所有信息无误后，点击「确认录入」批量写入数据库
- 入库结果反馈：成功录入 N 条交易记录

#### 3.2.7 验收标准

| # | 验收条件 | 说明 |
|---|---------|------|
| AC-1 | 支持上传至少 5 张图片并一次性识别 | 测试多图同时提交 |
| AC-2 | 支持粘贴剪贴板图片 | Ctrl+V / Cmd+V 粘贴截图后出现在图片列表中 |
| AC-3 | AI 返回结构化数据包含金额、描述、类型 | 验证 JSON 解析正确 |
| AC-4 | 已知描述自动匹配到正确分类 | 如"美团外卖"匹配"正餐" |
| AC-5 | 未识别分类的条目高亮显示 | 分类列为空时行背景/边框有视觉提示 |
| AC-6 | 所有条目分类不为空时才能提交 | 存在空分类时点击提交弹出错误提示 |
| AC-7 | 确认界面中每个字段可编辑 | 用户可修改类型、金额、分类、描述、操作人 |
| AC-8 | 批量入库后数据正确写入数据库 | 查看交易列表验证数据一致 |
| AC-9 | 无可用 AI 模型时给出明确提示 | 未配置任何 API Key 时引导用户到设置页 |
| AC-10 | AI 请求失败时展示错误信息并允许重试 | 网络超时、API 报错等场景 |

#### 3.4.1 筛选器与汇总卡片

##### 筛选器

页面顶部提供一行筛选控件，所有筛选条件变化时实时刷新下方所有区域（交叉统计表 + 图表 + 汇总卡片）：

| 筛选项 | 控件 | 默认值 | 说明 |
|--------|------|--------|------|
| 年份 | 年份选择器 | 当前年份 | 范围：数据库中最早交易年份 ~ 当前年份；无数据时仅显示当前年份；年度汇总视图下隐藏 |
| 类型 | 单选按钮组 | 消费 | 消费 / 收入 / 投资，三选一 |

- 筛选器状态不持久化，每次进入页面恢复默认值
- 类型为单选（交叉统计表同一时间只展示一种类型的分类体系）

##### 汇总卡片

筛选器下方展示一行汇总卡片（3 张），数据与当前筛选条件联动：

| 卡片 | 内容 | 说明 |
|------|------|------|
| 本月合计 | 当前年份 + 当前月份 + 当前类型的金额总计 | 若当前年份非今年，显示该年 12 月的数据 |
| 上月合计 | 上一个月同类型金额总计 | 与本月卡片的月份联动 |
| 年度合计 | 当前年份 + 当前类型的全年金额总计 | 即交叉统计表「合计」列的总计行值 |

- 金额显示使用千分号格式（与数据浏览页一致）
- 卡片下方显示环比变化（本月 vs 上月），正数显示为红色上箭头，负数显示为绿色下箭头，零不显示箭头
- 「本月」的定义：若筛选年份 = 当前年份，则取系统当前月份；若筛选年份为历史年份，则取 12 月

#### 3.4.2 交叉统计表（核心视图）

对标用户现有 Excel 透视表，以 Ant Design Table 实现。

##### 表结构

| 列 | 说明 | 宽度 |
|----|------|------|
| 分类 | 分类名称，固定在左侧 | 120px，固定列（`fixed: 'left'`） |
| 1月 ~ 12月 | 各月金额合计 | 等宽，可横向滚动 |
| 合计 | 全年金额总计 | 固定列（`fixed: 'right'`） |

- **行**：当前类型下所有**有数据的**分类，按分类排序顺序（`sort_order`）排列
- **末行**：总计行（每列求和），字体加粗
- **值**：金额显示千分号 + 两位小数；金额为 0 时显示 `—`（em dash），避免数字噪音
- **分类行只展示有数据的分类**：若某分类全年 12 个月金额均为 0，该行不显示（避免空行过多）

##### 数据查询

- 后端通过 `db:stats:cross-table` IPC 通道查询
- SQL 按 `(category_id, month)` 分组聚合，一次查询返回全年数据
- 年度汇总由后端计算后返回（非前端二次聚合），减少前端计算复杂度

##### 交互

- 表格支持横向滚动（月份列较多时），分类列和合计列固定
- 无分页（分类数量有限，通常 < 30 行）
- 不支持行内编辑（只读视图）

##### 空状态

- 当前筛选条件下无任何交易数据时，显示空状态占位图 + 提示文案「暂无数据，请先录入交易记录」

##### 图表数据下钻（查看明细）

趋势柱状图支持右键点击数据块，快速跳转至数据浏览页查看该数据块对应的明细交易记录。

**触发方式**
- 在月度趋势图或年度趋势图的任意数据块上**右键单击**
- 弹出上下文菜单，显示「查看明细」选项

**跳转逻辑**
点击「查看明细」后，应用自动导航至数据浏览页（`/`），并根据当前图表上下文自动设置以下筛选条件：

| 视图 | 自动设置的筛选条件 |
|------|-------------------|
| **月度趋势图** | 日期范围 = 该月份全月（如 2024-10-01 ~ 2024-10-31）<br>分类 = 右键点击的分类<br>类型 = 当前统计报表选中的类型（消费/收入/投资） |
| **年度趋势图** | 日期范围 = 该年份全年（如 2024-01-01 ~ 2024-12-31）<br>分类 = 右键点击的分类<br>类型 = 当前统计报表选中的类型（消费/收入/投资） |

**状态传递**
- 通过 URL query parameters 传递筛选状态（如 `/?dateFrom=2024-10-01&dateTo=2024-10-31&category_id=5&type=expense`）
- 数据浏览页解析 URL 参数，自动应用筛选条件并加载数据
- 用户可在数据浏览页手动清除或修改筛选条件

**来源标记与状态保留**
- 跳转时 URL 追加来源标记 `from=statistics`，标识是从统计页跳转而来
- 追加统计页当前状态参数：`year`（年份）、`tab`（视图Tab：monthly/yearly）、`soloCategory`（独显的分类，可选）
- 完整 URL 示例：`/?dateFrom=2024-10-01&dateTo=2024-10-31&category_id=5&type=expense&from=statistics&year=2024&tab=monthly&soloCategory=正餐`
- 数据浏览页检测到 `from=statistics` 时显示「← 返回统计页」按钮
- 返回按钮点击后导航至 `/statistics?year=2024&tab=monthly&soloCategory=正餐`，恢复统计页完整状态

**边界处理**
- 点击的块金额为 0 时：仍允许查看明细（结果为空）
- 分类为「合计」列（如点击总计行）：不设置分类筛选，仅按日期范围筛选
- 导航前若数据浏览页有未保存的编辑/新增行：提示用户保存或放弃

### 3.5 数据导入/导出

#### 3.5.1 导入

- **支持格式**：Excel（.xlsx）
- **兼容现有格式**：读取名为 "detail" 的工作表，固定列结构（日期、类型、金额、分组、描述、操作人）
- **导入模式**：全量覆盖（清空现有交易和操作人数据后重新导入）
- **导入流程**：
  1. 选择 Excel 文件
  2. 预览解析结果（展示有效记录数、操作人列表、分类列表、异常行提示）
  3. 用户确认后执行全量覆盖导入
- **分类处理**：自动匹配已有分类，未匹配的自动创建新分类
- **操作人处理**：清空后重建，根据导入数据自动创建
- **添加时间处理**：
  - 源文件包含「添加时间」列时，使用该时间作为 `created_at`，保持原始录入时间
  - 源文件不包含该列（旧格式，6列）时，以实际导入写入时间作为 `created_at`
  - 支持的时间格式：`YYYY-MM-DD HH:mm:ss`、Excel 日期时间序列号、纯日期（自动补全 `00:00:00`）

#### 3.5.2 导出

- **支持格式**：Excel（.xlsx）和 CSV（.csv），用户在导出时选择
- **导出列**：固定 7 列结构，包含添加时间：

| 列 | 内容 | 说明 |
|----|------|------|
| 日期 | `YYYY-MM-DD` | 交易日期 |
| 类型 | 消费 / 收入 / 投资 | 中文显示名 |
| 金额 | 数字（保留两位小数） | 负数原样导出（退款/冲正场景） |
| 分组 | 分类名称 | 列名为「分组」以兼容用户现有 Excel |
| 描述 | 自由文本 | 可为空 |
| 操作人 | 操作人名称 | 可为空 |
| **添加时间** | `YYYY-MM-DD HH:mm:ss` | **新增**，交易首次录入系统的时间 |

- **导出范围**：默认导出全部交易记录；支持按以下条件筛选后导出：
  - 日期范围（起止日期）
  - 交易类型（消费 / 收入 / 投资，可多选）
  - 分类（可多选）
  - 操作人（可多选）
  - 关键词（搜索描述字段）
- **排序**：导出数据按日期升序排列（与用户 Excel 习惯一致）
- **Excel 格式细节**：
  - 工作表名称为 `detail`（与导入格式一致，导出文件可直接作为导入源）
  - 首行为表头（日期、类型、金额、分组、描述、操作人、添加时间）
  - 添加时间列宽度 18 字符，时间格式按文本显示
- **CSV 格式细节**：
  - UTF-8 with BOM 编码（确保 Excel 打开时中文不乱码）
  - 首行为表头
  - 字段分隔符为逗号，含逗号或换行的字段用双引号包裹

##### 导出流程

1. 用户在设置页「数据管理」Tab 中点击「导出数据」按钮

   - 预览：显示符合条件的记录数
3. 点击「导出」按钮，弹出系统保存对话框
4. 默认文件名：`Moneta_导出_YYYYMMDD.xlsx`（或 `.csv`），日期为当天
5. 导出完成后提示：成功导出 N 条记录到 `文件路径`

### 3.7 AI 模型配置

#### 3.7.1 概述

AI 图片识别功能依赖外部多模态 AI API。系统内置固定的模型列表，用户选择模型并填入 API Key 即可使用。不支持用户自行添加自定义模型，新增模型通过应用版本更新提供。模型配置入口位于「设置」页面。

#### 3.7.2 内置模型列表

| 模型 | 平台 | API 格式 | Endpoint | 模型标识 |
|------|------|----------|----------|----------|
| GLM-4.5V | 智谱 AI (BigModel) | OpenAI 兼容 | `https://open.bigmodel.cn/api/paas/v4` | `glm-4.5v` |
| Doubao Seed 2.0 | 火山引擎 (ByteDance) | OpenAI 兼容 | `https://ark.cn-beijing.volces.com/api/v3` | 用户填入推理接入点 ID（如 `ep-xxx`） |

- Endpoint 默认为官方地址，用户**可修改**（如使用 SiliconFlow 等第三方平台调用同一模型）
- 模型标识默认为官方标识，用户**可修改**（如第三方平台的模型标识可能不同）
- 后续版本可扩展更多内置模型（如 Kimi、GPT-4o、Claude 等），需通过代码更新

#### 3.7.3 API Key 安全存储

- 使用 Electron `safeStorage` API 对 API Key 进行加密后存储到本地配置文件
- 设置界面中 API Key 以掩码形式显示（如 `****abcd`），仅在编辑时可查看完整值
- API Key 不会以明文形式写入任何日志或配置文件

#### 3.7.4 模型切换与默认设置

- 当内置模型列表中仅有一个模型时，自动作为默认模型，无需用户手动选择
- 当内置模型列表中有多个模型时：
  - 用户可在设置页面指定一个**默认模型**
  - 在 AI 识别页面，用户可通过下拉框**临时切换**模型（不改变默认设置）
- 未配置任何 API Key 时，AI 识别页面显示引导提示，引导用户前往设置页配置

#### 3.7.5 连通性测试

- 每个模型配置 API Key 后，提供「测试连接」按钮
- 测试逻辑：发送一个最小化的 API 请求（如发送纯文本 prompt），验证 API Key 是否有效
- 测试结果：成功（显示绿色对勾 + 模型名称）或失败（显示错误原因，如认证失败、网络不可达）

#### 3.7.6 设置页 UI

- 以卡片形式展示内置模型列表
- 每个模型卡片显示：模型名称、平台名称、配置状态（已配置/未配置）、是否为默认
- 用户操作：填写/编辑 API Key、修改 Endpoint、修改模型标识、测试连接、设为默认
- 不提供删除或新增模型的入口

### 3.8 应用锁屏与 PIN 码解锁

#### 3.8.1 概述

为保护用户财务数据隐私，Moneta 提供应用锁屏功能。用户设置 PIN 码后，每次启动应用默认处于锁屏状态，需输入正确的 PIN 码后才能进入主界面使用。

#### 3.8.2 首次使用

- 应用首次启动时（未设置过 PIN 码），弹出引导界面要求用户设置 PIN 码
- PIN 码为固定 **4 位数字**（0-9）
- 设置流程：输入 PIN → 再次输入确认 → 两次一致后保存
- 两次输入不一致时提示错误，要求重新输入
- PIN 码设置完成后立即生效，下次启动将进入锁屏

#### 3.8.3 锁屏界面

- 应用启动后首先展示锁屏界面，覆盖整个窗口
- 界面元素：
  - 应用名称（Moneta）和图标
  - 4 位 PIN 码输入区域（每位独立输入框，输入时显示圆点掩码）
  - 输入完成后自动验证（无需点击确认按钮）
  - 错误提示区域
- 锁屏状态下不可访问应用任何功能，不可通过快捷键或其他方式绕过

#### 3.8.4 解锁流程

1. 用户在锁屏界面输入 4 位 PIN 码
2. 输入第 4 位后自动触发验证
3. 验证成功：锁屏界面消失，进入应用主界面
4. 验证失败：
   - 输入框清空并抖动，提示「PIN 码错误，请重试」
   - 显示剩余可尝试次数

#### 3.8.5 安全机制

- **PIN 码加密存储**：复用 Electron `safeStorage` API 加密后存储到本地配置文件，与 API Key 存储方式一致
- **错误次数限制**：连续输入错误 5 次后锁定应用，需等待 30 秒后才能继续尝试
- **锁定提示**：锁定期间显示倒计时，提示剩余等待时间

#### 3.8.6 PIN 码管理

PIN 码管理入口位于「设置」页面：

- **修改 PIN 码**：需先输入当前 PIN 码验证身份，再输入新 PIN 码（两次确认）
- **PIN 码始终启用**：不提供关闭锁屏的选项，PIN 码一旦设置即持续生效

#### 3.8.7 验收标准

| # | 验收条件 | 说明 |
|---|---------|------|
| AC-L1 | 首次启动弹出 PIN 码设置引导 | 未设置 PIN 时强制引导设置 |
| AC-L2 | PIN 码固定 4 位数字 | 仅接受 0-9 数字输入 |
| AC-L3 | 设置 PIN 需两次确认一致 | 两次不一致时提示错误 |
| AC-L4 | 每次启动应用显示锁屏界面 | 主界面在解锁前不可见、不可操作 |
| AC-L5 | 输入正确 PIN 后解锁进入主界面 | 验证成功后锁屏消失 |
| AC-L6 | 输入错误 PIN 后提示并清空 | 显示错误信息和剩余次数 |
| AC-L7 | 连续 5 次错误后锁定 30 秒 | 锁定期间显示倒计时 |
| AC-L8 | PIN 码加密存储 | 使用 safeStorage API，配置文件中不含明文 |
| AC-L9 | 设置页可修改 PIN 码 | 需验证旧 PIN 后才能设置新 PIN |
| AC-L10 | 锁屏界面输入掩码显示 | 输入数字显示为圆点 |
| AC-L11 | 输入 4 位后自动验证 | 无需手动点击确认按钮 |

### 3.9 多语言

#### 3.9.1 功能概述

Moneta 支持多语言界面，使应用能够服务不同语言的用户。用户可在设置页切换界面语言，应用立即生效无需重启。

#### 3.9.2 支持的语言

**初期（v0.6）**：
- 简体中文（zh-CN）
- 英文（en-US）

**未来扩展**：
- 繁体中文（zh-TW）
- 日语（ja-JP）
- 韩语（ko-KR）

#### 3.9.3 语言切换

**位置**：设置页新增「语言」Tab，位于「MCP 配置」和「安全设置」之间

**界面设计**：
- 使用 Radio.Group 单选组件
- 选项：「简体中文」和「English」
- 点击「保存」按钮后生效
- 保存后界面立即切换，无需重启应用

**配置存储**：
- 保存到用户本地配置文件（`app.getPath('userData')/config.json`）
- 新增 `language` 字段，值为 `zh-CN` 或 `en-US`

#### 3.9.4 默认语言检测

**首次启动时**：
- 自动检测系统语言（`app.getLocale()` 或 `navigator.language`）
- 系统语言为 `zh-*` 时，默认使用简体中文
- 其他情况默认使用英文
- 用户可随时在设置页切换

**后续启动时**：
- 读取配置文件中的 `language` 字段
- 无配置时回退到系统语言检测

#### 3.9.5 翻译范围

**需要翻译的内容**：
- 所有 UI 文本（按钮、标签、菜单项、占位符、标题）
- 消息提示（成功/错误/警告 Toast）
- Modal 对话框内容
- 表格列标题
- 表单验证消息
- 日期时间格式文本
- 交易类型的显示名称（`expense` → "消费"/"Expense"）

**不翻译的内容**（用户数据保持原始输入）：
- 分类名称（用户创建时使用的语言）
- 分类的 AI 描述字段
- 操作人姓名
- 交易描述
- Ant Design 组件自带文本（通过 `ConfigProvider` locale 自动切换）
- dayjs 时间文本（通过 `dayjs.locale()` 自动切换）

**设计原则**：
- 用户数据（分类、操作人、描述）是用户创建的内容，不应被系统翻译
- 中文用户可以创建中文分类，英文用户可以创建英文分类
- 仅 UI 元素根据用户选择的语言动态切换

#### 3.9.6 系统预置分类的处理

**初期方案（v0.6）**：

### 3.10 数据库加密

#### 3.10.1 功能概述

为保护用户财务数据隐私，Moneta 使用 SQLCipher（通过 `better-sqlite3-multiple-ciphers`）对 SQLite 数据库进行透明加密。数据库文件在磁盘上始终处于加密状态，即使应用运行期间也无法被第三方工具直接读取。

#### 3.10.2 设计决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 加密方案 | SQLCipher 数据库级加密 | 数据始终加密（含 WAL 日志），查询/索引/搜索完全不受影响 |
| 密钥管理 | OS 级自动加密（safeStorage） | 用户无感知，无需额外密码；macOS Keychain / Windows DPAPI 保护 |
| 密钥派生 | 32 字节随机密钥 | 首次启动自动生成，通过 safeStorage 加密后存入 config.json |
| 加密算法 | sqleet（ChaCha20） | SQLite3MultipleCiphers 默认算法，性能优于 AES-256-CBC |
| 依赖替换 | `better-sqlite3` → `better-sqlite3-multiple-ciphers` | API 完全兼容的 fork，维护活跃（v12.8.0），支持 Electron 33 |

#### 3.10.3 密钥管理

**密钥生成与存储**：
- 首次启动时，使用 `crypto.randomBytes(32)` 生成 32 字节（256 位）随机密钥
- 将密钥转为 hex 字符串，通过现有的 `encryptString()` 函数（基于 Electron `safeStorage`）加密
- 加密后的密钥存储在 `config.json` 的 `dbKeyEncrypted` 字段中，与 `pinEncrypted`、`apiKeyEncrypted` 同级
- 后续启动时，通过 `decryptString()` 解密还原密钥，用于打开数据库

**安全等级**：
- 与现有 PIN 码、API Key 存储机制一致，依赖 OS 凭据系统
- macOS：Keychain 保护，跨机器无法解密
- Windows：DPAPI 保护，仅当前 Windows 用户可解密
- `safeStorage` 不可用时降级为 base64（与现有行为一致，打印警告日志）

#### 3.10.4 数据库连接变更

打开数据库时设置加密密钥和参数：

```
1. new Database(dbPath)
2. PRAGMA key = "x'<hex_key>'"
3. PRAGMA cipher = 'sqleet'
4. PRAGMA cipher_page_size = 4096
5. PRAGMA kdf_iter = 256000
6. PRAGMA journal_mode = WAL       （现有）
7. PRAGMA foreign_keys = ON        （现有）
```

密钥设置必须在其他所有操作之前完成。

#### 3.10.5 现有用户数据迁移

现有用户的数据库为明文格式，升级后需自动迁移为加密格式。

**迁移策略**：SQLCipher 原生 `sqlcipher_export` + `ATTACH DATABASE`。

**迁移流程**：

1. **检测**：启动时检查 `config.json` 是否有 `dbKeyEncrypted` 字段
   - 无 → 执行迁移（旧用户升级或新安装）
   - 有 → 正常打开加密数据库

2. **迁移步骤**：
   - 以明文模式打开现有 `moneta.db`
   - 生成 32 字节随机密钥，加密存入 `config.json`
   - 将原数据库备份为 `moneta.db.plain.bak`
   - `ATTACH DATABASE` 创建加密临时数据库 `moneta.db.enc`
   - 执行 `SELECT sqlcipher_export('moneta.db.enc')` 导出全部数据
   - 验证导出完整性（检查表数、行数）
   - 关闭连接，用加密数据库替换原文件（删除 WAL/SHM 文件）
   - 删除备份文件

3. **回退机制**：
   - 迁移前备份原数据库为 `moneta.db.plain.bak`
   - 检测到 `.bak` 文件存在且加密数据库不可读时，优先恢复备份
   - 迁移成功并验证通过后删除备份

4. **边界处理**：
   - **空数据库**（新安装）：直接创建加密数据库，跳过迁移
   - **迁移中断**（崩溃/断电）：检测到 `.bak` 文件时恢复备份重新迁移
   - **MCP Server**：迁移过程中 HTTP Server 尚未启动，不受影响

#### 3.10.6 MCP 兼容性

MCP Server 不直接访问数据库文件，仅通过 HTTP 与主应用通信。主应用中的 MCP HTTP Server 使用与所有 IPC Handler 相同的 `getDatabase()` 连接。因此数据库加密对 MCP 功能完全透明，无需任何改动。

#### 3.10.7 安全性分析

| 攻击场景 | 当前状态 | 加密后 |
|---------|---------|--------|
| 直接拷贝 .db 文件查看 | 明文可读 | ChaCha20 加密，无法读取 |
| DB Browser 等工具打开 | 可直接查看 | 需要密钥才能打开 |
| 磁盘取证 | 明文数据可恢复 | 磁盘上的数据始终加密（含 WAL） |
| 运行时内存提取 | 数据在内存中明文 | 同左（不可避免，需 OS 级保护） |
| 同机已登录攻击者 | 可直接读文件 | 需要 safeStorage 凭据 |
| 跨机器拷贝 | 完全可读 | 密钥与 OS 用户绑定，跨机器无法解密 |

**不防护的场景**：应用运行时内存中的明文数据、恶意软件内存转储、用户主动导出的 Excel/CSV 文件。

#### 3.10.8 改动范围

| 文件 | 改动 | 说明 |
|------|------|------|
| `package.json` | 替换依赖 | `better-sqlite3` → `better-sqlite3-multiple-ciphers` |
| `src/main/database/connection.ts` | 核心改动 | 添加密钥解密 + PRAGMA key 设置 + 明文→加密迁移逻辑 |
| `src/main/services/config.service.ts` | 小改 | `loadConfig()` 增加 `dbKeyEncrypted` 字段处理 |

| MCP Server (`src/mcp/`) | 无改动 | 通过 HTTP 访问，加密透明 |
| 所有 Repository 文件 | 无改动 | 通过 `getDatabase()` 透明访问 |
| 所有 IPC Handler | 无改动 | 通过 `getDatabase()` 透明访问 |
| 前端代码 | 无改动 | 加密完全在主进程，渲染进程无感知 |

#### 3.10.9 验收标准

| # | 验收条件 | 说明 |

#### 5.6.1 问题背景

在 AI 图片识别确认页面和 MCP 导入确认页面中，用户可能需要花费较长时间进行以下操作：
- 逐条核对和修改识别结果
- 补充未匹配的分类
- 插入遗漏的交易条目
- 调整金额、描述等字段

当前实现中，如果用户误操作关闭页面、程序崩溃或需要临时退出应用，所有编辑进度都会丢失，需要重新开始整个导入流程，体验较差。

#### 5.6.2 功能目标

为导入确认页面提供**自动草稿保存**能力：
- 用户在确认页面的所有编辑操作自动持久化到本地
- 即使关闭页面、重启应用，也能恢复未完成的导入草稿
- AI 识别和 MCP 导入共享一份草稿存储，简化设计
- 草稿成功导入后自动清理

#### 5.6.3 草稿数据范围

**全局唯一草稿**：同一时间只保存一份草稿，不区分 AI 识别或 MCP 导入。

**草稿内容**：
- 待导入交易记录数组（含所有字段修改）
- 草稿来源标识（`ai` 或 `mcp`）
- 草稿元数据（创建时间、最后更新时间）
- **AI 识别特有**：记账日期、图片文件路径列表
- **MCP 特有**：数据来源描述（source）

**不保存的内容**：
- 临时 UI 状态（如滚动位置、展开的折叠面板）
- 撤销/重做历史栈

#### 5.6.4 草稿生命周期

| 阶段 | 行为 | 触发条件 |
|------|------|---------|
| **创建** | 进入确认页面时创建草稿 | 识别/导入成功返回数据后 |
| **更新** | 自动保存编辑内容 | 字段变更、增删行、切换操作人等 |
| **恢复** | 用户通过系统提示恢复草稿 | 点击系统界面上的「继续未完成导入」提示 |
| **覆盖** | 新数据覆盖旧草稿 | 开始新的导入（AI 或 MCP）时，若用户未恢复旧草稿 |
| **提交** | 确认入库后删除草稿 | 用户点击「确认录入」且全部成功 |
| **放弃** | 用户主动放弃草稿 | 点击系统提示中的「放弃」或确认页面的「放弃导入」 |

#### 5.6.5 草稿存储方案

- **存储位置**：SQLite 数据库独立表 `import_draft`（单条记录）
- **数据结构**：JSON 序列化存储，字段包括：
  - `id`: 固定值 `current`（单行表）
  - `source`: 来源标识（`ai` 或 `mcp`）
  - `data`: 草稿数据 JSON
  - `created_at`: 创建时间
  - `updated_at`: 最后更新时间
- **并发处理**：单用户应用，直接覆盖写入，始终只保留一份草稿

**显示位置**：数据浏览页（`/`）表格上方、记账日期工具栏下方

**视觉设计**：
- 使用 Alert 或 Card 组件，占据整行宽度

- 左侧 4px 橙色边框强调
- 包含图标（⚠️ 或 📝）、文字信息、操作按钮

**显示逻辑**：
- **无草稿时**：不显示任何提示，页面保持原有布局
- **有草稿时**：显示提示卡片，包含：
  - 草稿来源标识（AI 识别 / MCP 导入）

  - 操作按钮：「继续导入」、「放弃」

**页面层级**：
```
数据浏览页
├── 页面标题栏
├── 草稿提示卡片（有草稿时显示）
├── 记账日期工具栏（日期选择器 + 导入/新增按钮）
└── 交易数据表格
```

**交互流程**：
1. 用户点击「继续导入」
2. 根据草稿来源跳转到对应页面：
   - `source: 'ai'` → `/ai-recognition`
   - `source: 'mcp'` → `/mcp-import`
3. 加载草稿数据，用户继续编辑

**放弃流程**：
1. 用户点击「放弃」
2. 二次确认对话框：「确定放弃未完成的导入草稿？此操作不可恢复」
3. 确认后删除草稿，提示消失

> 注：MCP 草稿在 AI 识别页面恢复后，原 MCP 特有的 `source` 字段保留但不展示，若用户确认导入则正常入库。

#### 5.6.7 新导入覆盖旧草稿

**策略：用户开始新导入时，若未处理旧草稿，直接覆盖**

**场景**：
- 用户点击「图片识别导入」按钮（无论系统提示是否存在）
- AI 助手调用 `send_transactions` 发送新数据

**处理逻辑**：
1. 检测是否存在未完成草稿
2. 若存在且用户未通过系统提示恢复：
   - 直接用新数据覆盖旧草稿
   - 页面顶部显示弱提示：「已开始新的导入，之前的草稿已替换」
3. 若用户已通过系统提示恢复并正在编辑：正常创建新草稿（同一会话内不允许同时存在两个导入流程）

**原因**：
- 用户主动开始新导入，说明当前意图是处理新数据
- 若用户在乎旧草稿，会通过系统提示先恢复
- 不强制弹窗询问，减少打断

#### 5.6.8 自动保存策略

- **响应式即时保存**：通过 `useEffect` 监听 `rows`、`defaultOperatorId`、`accountingDate` 状态变化，任何编辑操作在 React 状态更新后立即触发保存，无防抖延迟
- **无丢失风险**：无论用户通过返回按钮、侧栏导航还是关闭应用退出，编辑数据都已持久化
- **批量编辑优化**：全量替换保存，不记录变更日志

#### 5.6.9 文件引用处理（AI 草稿）

AI 草稿中保存的是图片文件的绝对路径。恢复时：
- 检查每个路径的文件是否存在
- **全部存在**：正常恢复，显示缩略图列表
- **部分缺失**：提示「X 张原图片已丢失」，保留草稿数据但清空图片列表，用户可重新上传图片
- **全部缺失**：同部分缺失处理，提示用户图片丢失

> MCP 草稿不涉及文件引用，无需此处理。

#### 5.6.10 异常处理

| 场景 | 处理方式 |
|------|---------|
| 草稿数据解析失败 | 视为无效草稿，提示「草稿数据损坏，请重新开始」并删除 |
| 草稿版本不兼容（旧版本格式） | 尝试迁移，失败则提示并删除 |
| 存储空间不足 | 静默失败，不影响用户编辑，下次启动时尝试重新保存 |
| MCP 连续调用覆盖 | 见 5.6.7 边界情况处理，以最后一次数据为准 |

## 6. MCP 账单导入助手（v0.5）

### 6.1 功能概述

MCP（Model Context Protocol）账单导入助手允许用户通过 AI 助手（如 Claude）直接与应用交互，分析本地 Excel/CSV 文件内容并智能导入交易数据。用户无需手动处理 Excel 格式转换，只需将文件路径提供给 AI，AI 将自动分析数据结构、识别字段映射，并生成符合 Moneta 格式的交易记录供用户确认后入库。

### 6.2 使用场景

1. **历史账单迁移**：用户有多年前不同格式的 Excel 记账文件，通过 AI 助手描述文件结构，AI 自动解析并映射到 Moneta 字段
2. **第三方导出文件导入**：用户从银行 App、支付宝、微信等导出的账单 CSV，AI 自动识别日期格式、金额正负含义、交易类型等
3. **复杂数据清洗**：原始数据包含冗余列、混合多种交易类型，AI 辅助智能分类和清洗
4. **批量格式转换**：多个不同格式的旧账单文件，一次性通过 AI 分析并统一导入

### 6.3 功能架构

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   用户对话界面   │ ←→  │   AI 助手        │ ←→  │  MCP Server     │
│  (Claude 等)    │     │  (Claude Desktop)│     │  (Moneta 内置)  │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                                                        │
                        ┌─────────────────┐            │
                        │   数据确认界面   │ ←─────────┘
                        │ (类似 AI 识别页) │
                        └─────────────────┘
                                │
                                ↓
                        ┌─────────────────┐
                        │   批量写入数据库 │
                        └─────────────────┘
```

### 6.4 MCP Server 能力

Moneta 应用内置 MCP Server，通过 **stdio 模式** 与 Claude Desktop 通信。暴露以下工具（Tools）供 AI 调用。

**假设**：AI 助手（如 Claude Desktop）能够直接读取用户本地文件，MCP 只负责「接收转换好的数据并展示确认界面」这一环节。

#### 6.4.0 MCP Server 注册方式

**推荐方式：应用内一键配置（半自动）**

Moneta 设置页「AI 模型」Tab 中提供「配置 Claude Desktop MCP」按钮：

1. 用户点击按钮
2. Moneta 自动检测 Claude Desktop 配置文件位置：
   - macOS: `~/Library/Application Support/Claude/claude_desktop_config.json`
   - Windows: `%APPDATA%/Claude/claude_desktop_config.json`
3. Moneta 将 MCP Server 配置写入该文件：

```json
{
  "mcpServers": {
    "moneta": {
      "command": "/Applications/Moneta.app/Contents/MacOS/moneta-mcp",
      "args": ["--mcp"],
      "env": {}
    }
  }
}
```

4. 提示用户重启 Claude Desktop 以生效

**备用方式：手动配置**

如果自动配置失败或用户偏好手动，提供配置指引：

```markdown
1. 打开 Claude Desktop → Settings → Developer → Edit Config
2. 在 claude_desktop_config.json 中添加：

{
  "mcpServers": {
    "moneta": {
      "command": "<Moneta安装路径>/moneta-mcp",
      "args": ["--mcp"]
    }
  }
}

3. 重启 Claude Desktop
```

**MCP Server 启动模式**

- Moneta 主应用与 MCP Server 分离，避免 stdio 通信阻塞主进程
- 打包时单独输出 `moneta-mcp` 可执行文件（或 Node.js 脚本）
- `moneta-mcp --mcp` 启动时，通过 stdio 与 Claude Desktop 通信，支持 MCP 协议
- MCP Server 需要与主应用通信获取分类、操作人等数据，可通过本地 HTTP 或文件 socket

#### 6.4.1 工具列表

| 工具名称 | 功能 | 参数 |
|---------|------|------|
| `get_categories` | 获取当前分类列表（供 AI 智能匹配） | `type`: 可选，筛选特定类型（expense/income/investment） |
| `get_operators` | 获取现有操作人列表 | - |
| `send_transactions` | AI 将转换好的交易数据发送给 Moneta | `transactions`: 交易记录数组（见下方格式）<br>`source`: 数据来源描述（如 "招商银行账单 2024-01"） |

每个工具的 `description` 字段应清晰说明用途和使用场景，供 AI 助手理解何时调用。详见 §6.6 MCP Skill 文档。

```typescript
    "name": "正餐",
    "type": "expense",
    "description": "外卖、堂食、食堂、餐厅",  // AI 描述字段，辅助 AI 理解分类含义
    "sort_order": 1
  }
]
```

> **注意**：`description` 字段是分类的「AI 描述」，用于帮助 AI 助手更准确地匹配分类。如 "外卖、堂食、食堂、餐厅" 提示 AI 看到外卖订单时应匹配到「正餐」分类。

**`get_operators` 返回格式**：

```typescript
[
  {
    "id": 1,
    "name": "本人"
  }
]
```

#### 6.4.3 交易数据格式

AI 调用 `send_transactions` 时，`transactions` 数组每项格式如下：

```typescript
{
  date: string;           // 日期，格式 YYYY-MM-DD
  type: "expense" | "income" | "investment";
  amount: number;         // 金额，正数
  category_id?: number;   // 分类 ID（AI 可提前匹配，也可留空让用户选择）
  category_name?: string; // 分类名称（用于显示，category_id 为空时 AI 可建议）
  description: string;    // 描述
  operator_id?: number;   // 操作人 ID（可选）
  operator_name?: string; // 操作人名称（可选，用于显示）
}
```

**说明**：
- AI 读取文件后自行完成字段映射、日期格式转换、金额解析、分类匹配
- `category_id` 和 `operator_id` 可通过 `get_categories` / `get_operators` 获取后匹配
- AI 匹配分类时，应结合 `name` 和 `description` 字段进行语义匹配
- 如果 AI 不确定分类，可以只填 `category_name` 作为建议，或留空由用户在确认界面选择

### 6.5 导入流程

#### 6.5.1 AI 处理阶段（在 AI 助手端完成）

1. 用户提供文件路径和内容给 AI 助手（通过对话粘贴或 AI 直接读取本地文件）
2. AI 分析文件内容，完成以下转换：
   - 解析日期字段并统一格式化为 `YYYY-MM-DD`
   - 解析金额（处理正负号、货币符号、千分号等）
   - 判断交易类型（收入/支出/投资）
   - 调用 `get_categories` 获取分类列表，智能匹配分类（如 "美团外卖" → "正餐"）
   - 组装描述文本
3. AI 生成符合 Moneta 格式的交易记录数组

#### 6.5.2 发送数据阶段

AI 调用 `send_transactions` 工具，将转换好的数据发送给 Moneta：

```json
{
  "transactions": [
    {
      "date": "2024-01-15",
      "type": "expense",
      "amount": 128.50,
      "category_name": "正餐",
      "description": "美团外卖 - 午餐",
      "operator_name": "本人"
    }
  ],
  "source": "支付宝账单 2024年1月"
}
```

#### 6.5.3 预览确认阶段

1. Moneta 收到数据后，自动打开「MCP 导入确认界面」（复用 AI 图片识别确认页，路由 `/mcp-import`）
2. 界面展示：
   - 数据来源（`source` 字段）
   - 共 N 条待导入记录
   - 分类已匹配 M 条，待确认 P 条
3. 用户在界面中完成最终编辑：
   - 修改任意字段
   - 删除不需要的行
   - 插入新行
   - 为未匹配分类的行选择分类（红色高亮提示）

#### 6.5.4 执行导入阶段

1. 用户点击「确认导入」后，数据批量写入数据库
2. 显示导入结果：成功 N 条
3. 界面自动关闭或返回数据浏览页

### 6.6 数据确认界面

复用 AI 图片识别的确认界面组件，显示以下信息：

| 元素 | 说明 |
|------|------|
| 导入摘要 | 共 N 条，待补充分类 M 条，已智能匹配分类 P 条 |
| 全局操作人 | **初始为空**，用户必须选择后才能提交；选择后所有行同步更新 |
| 记账日期 | **初始为空**（即使 MCP 数据携带日期也清空），用户必须手动确认日期后才能提交 |
| 可编辑表格 | 同 AI 识别确认页，支持行内编辑、插入、删除 |
| 分类高亮 | 未匹配到分类的行红色高亮 |
| 提交校验 | 分类为空、日期未选择、操作人未选择时均阻止提交 |

### 6.7 与 AI 图片识别的差异

| 维度 | AI 图片识别 | MCP 账单导入 |
|------|------------|-------------|
| 数据来源 | 图片（截图/照片） | 本地 Excel/CSV 文件 |
| AI 角色 | 提取图片中的交易信息 | 分析文件结构、智能映射字段 |
| 识别精度 | 依赖图片质量和 OCR | 依赖文件结构化程度 |
| 分类匹配 | AI 根据描述建议分类 | AI 根据规则映射 + 智能匹配 |
| 进入方式 | 数据浏览页「图片识别导入」按钮 | AI 助手主动调用 MCP 触发 |
| 确认界面 | 独立的 `/ai-recognition` 页面 | 复用相同组件，路由 `/mcp-import` |

### 6.8 安全与权限

1. **文件访问白名单**：MCP Server 仅允许访问用户明确指定的文件路径（通过对话框选择后传入）
2. **执行确认**：`execute_import` 调用前必须通过确认界面获得用户明确授权
3. **沙箱限制**：MCP Server 仅暴露必要的只读和受控写入接口
4. **日志记录**：所有 MCP 调用记录到应用日志，便于审计

### 6.6 MCP Skill 文档（供 AI 助手使用）

为了让 AI 助手（如 Claude）能够正确、高效地使用 MCP 工具帮助用户导入账单，需要提供一份 Skill 文档。该文档通常放在项目的 `.claude/skills/` 或 `docs/skills/` 目录下，AI 助手可通过读取该文档学习最佳实践。

#### Skill 文档建议位置

```
.claude/skills/mcp-moneta-import.md
```

或

```
docs/skills/mcp-moneta-import.md
```

#### Skill 文档核心内容

**1. 工具使用指南**

```markdown
# Moneta 账单导入助手

## 何时使用
当用户想要从 Excel/CSV 文件导入账单数据到 Moneta 记账软件时使用本技能。

## 工具使用流程

### 步骤 1：获取元数据
调用 `get_categories()` 获取所有分类（含 AI 描述）。
调用 `get_operators()` 获取操作人列表。

### 步骤 2：分析用户数据
用户会提供文件路径或直接粘贴内容。你需要：
1. 识别日期列并统一转换为 YYYY-MM-DD 格式
2. 识别金额列，统一转为正数，根据类型判断收/支
3. 识别描述/商品名称列，用于匹配分类
4. 如有交易类型列（收入/支出），映射为 expense/income

### 步骤 3：智能分类匹配
对于每条记录，结合分类的 name 和 description 进行匹配：
- "美团外卖" → 匹配「正餐」（description 包含"外卖"）
- "滴滴出行" → 匹配「交通」（description 包含"打车"）
- 无法确定时留空 category_id，只保留原始描述

### 步骤 4：发送数据
调用 `send_transactions({
  transactions: [...],
  source: "描述数据来源"
})`
Moneta 会自动打开确认界面，用户完成最终编辑后入库。

## 注意事项
- 金额必须是正数，类型字段判断收/支
- 日期格式严格为 YYYY-MM-DD
- 不确定的分类不要强行匹配，留给用户确认界面处理
- 复杂文件（如多 sheet、非常规格式）先询问用户确认字段映射
```
